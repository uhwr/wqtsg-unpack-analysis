### 2.2 HTTP 封装逻辑

`utils/http.js` 提供：

```text
get(url, data, loading = true)
post(url, data, loading = true)
query(url, query, variables, loading = true)
conv(response)
apiUrl()
```

`query()` 最终调用 `post()`：

```text
post(apiUrl, { query, variables })
```

响应转换规则：

```text
如果响应不是 object：返回 { code: 1, msg: "后端数据返回错误" }
如果响应包含 errors：返回 errors[0]
否则返回 { code: 0, msg: "", data: response.data }
```

因此页面通常通过 `code === 0` 判断接口成功。

登录页成功后：


### 7.3 签到请求

签到前先请求服务器时间：

```text

```

然后使用内置 RSA 公钥对服务器时间加密，生成 `pass`。

可见公钥类型：

```text
-----BEGIN PUBLIC KEY----- ... -----END PUBLIC KEY-----
```

随后提交：

```text
POST https://wechat.v2.traceint.com/index.php/wxApp/sign.html
```

提交数据包括：

```text

devices     base64(JSON.stringify(发现的 Beacon 设备))
location    base64(JSON.stringify(定位信息))
pass        RSA 加密后的服务器时间
```

## 9. 潜在风险点

以下是基于客户端解包代码的静态观察，不代表服务端必然存在漏洞，需在授权范围内结合服务端行为验证。

### 9.1 接口结构完全暴露

客户端包含完整 GraphQL 查询、字段名、变量名和业务操作名。攻击者拿到包后可以清楚知道：

```text
哪些接口存在
哪些参数可传
哪些操作是高权限操作
返回字段有哪些
```

风险缓解重点在服务端：

```text
每个 GraphQL resolver 必须做身份、角色、学校范围、资源范围校验。
```

### 9.2 token 存储在本地 storage

登录态保存在：

```text
wx.setStorageSync("userInfo", ...)
```

其中包含：

```text
getToken.accessToken
adm_account
```

且客户端会将 token 改写为：

```text
adm_account|accessToken
```

风险：如果终端环境、调试环境或备份数据被读取，token 可能泄露。

建议：

```text
服务端设置较短 token 生命周期
支持 token 失效和吊销
对高危操作二次校验或审计
绑定管理员、学校、设备或会话上下文
```

### 9.3 客户端不能作为权限边界

客户端包含页面跳转、按钮显示、学校选择等逻辑，但这些都可以被篡改或重放。尤其是：

```text
batMark
batFree
removeBlack
sendMsg
setQrCodeExpire
answer
```

这些操作必须在服务端验证：

```text
管理员是否有该学校权限
管理员是否有该操作权限
操作对象是否属于授权学校/馆区
批量 seat_keys 是否全部在授权范围内
```

### 9.4 GraphQL 查询使用 query 承载写操作

多个有副作用的操作使用 GraphQL `query` 字符串承载，例如：

```text
sendMsg
setfree
release
uploadProof
removeProof
answer
removeBlack
setQrCodeExpire
```

这不一定是漏洞，但会带来语义和中间层策略风险：

```text
缓存、审计、WAF、日志分类可能误判为读操作
安全策略应按字段级操作名判断，而不是只看 query/mutation 类型
```

### 9.5 批量座位操作风险较高

`batMark` / `batFree` 通过 `seat_keys` 批量操作：

```text
lib_id
seat_keys
proofId
msg
```

重点验证：

```text
seat_keys 数量限制
seat_keys 是否属于 lib_id
lib_id 是否属于管理员授权学校
重复 seat_keys 行为
越权 school/lib/seat 操作
服务端是否记录操作者和原因
```

### 9.6 通知发送接口风险较高

`sendMsg` 参数：

```text
endTime
title
content
userMobile
schId
showType
```

重点验证：

```text
是否限制发送范围
是否校验 schId 权限
是否限制 userMobile 批量/空值语义
是否有频率限制
是否有内容安全和长度限制
是否有操作审计
```

### 9.7 黑名单移除仅使用 userMobile

移除黑名单接口参数只有：

```text
userMobile
```

重点验证：

```text
是否结合学校范围校验
手机号是否唯一跨学校
管理员是否能移除其他学校黑名单
```

### 9.8 签到接口依赖客户端采集的设备和定位信息

签到页提交：

```text
devices = base64(JSON.stringify(Beacon 信息))
location = base64(JSON.stringify(定位信息))
pass = RSA(服务器时间)
```

注意：

```text
base64 不是安全保护，只是编码
公钥在客户端公开是正常现象
客户端采集到的 devices/location 不能被服务端盲信
```

服务端需要验证：

```text
pass 是否基于最近服务器时间
请求是否绑定授权参数 t
Beacon 是否属于该学校
定位是否在合理范围
时间窗口和重放限制
同一用户/设备的频率限制
```

### 9.9 图片凭证上传

`uploadProof` 接收：

```text
base64code
```

重点验证：

```text
文件大小限制
图片格式校验
恶意文件伪装
存储路径安全
内容审核
上传频率限制
```

## 10. 建议的下一步验证清单

在合法授权前提下，建议后续按以下顺序深入：

1. 将 `app-service.js` 拆分为模块文件并格式化，方便逐页审计。
2. 对 `service/dataService.js` 的每个 GraphQL 操作建立接口测试用例。
3. 验证 token 失效、403、过期、篡改时的服务端行为。
4. 验证学校越权：管理员 A 是否能访问/操作学校 B 的资源。
5. 验证馆区越权：`lib_id` 与 `sch_id` 不匹配时服务端是否拒绝。
6. 验证座位越权：`seat_keys` 是否能跨馆区或跨学校操作。
7. 验证批量操作数量限制和频率限制。
8. 验证通知发送的范围、频率、内容和审计。
9. 验证黑名单移除是否只凭手机号导致跨学校影响。
10. 验证签到接口是否能抵抗重放、伪造设备、伪造定位。

## 11. 当前结论

该小程序管理端的前端代码清晰暴露了管理后台的主要能力和接口字段。客户端本身没有发现可直接证明服务端漏洞的证据，但存在多个必须由服务端强制保护的高权限操作面。后续深入应以“服务端鉴权、资源归属校验、批量操作限制、通知发送限制、签到防伪造”为重点。
