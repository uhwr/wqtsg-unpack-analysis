## 4. 蓝牙签到整体流程

整体流程如下：

```text
进入 pages/blue/blue
  ↓
checkEnv 检查蓝牙和定位权限
  ↓
如果页面 URL 中没有 t 参数
  ↓
跳转 pages/blue/auth 授权页
  ↓
授权后带 t 参数回到 blue 页面
  ↓
POST /index.php/wxApp/devices.html
  ↓
服务端根据 t 返回当前学校允许的 Beacon UUID 列表
  ↓
客户端获取 GPS 定位
  ↓
客户端按服务端返回的 UUID 搜索附近 Beacon
  ↓
用户点击签到
  ↓
GET /index.php/wxApp/getTime.html 获取服务器时间
  ↓
使用 wxmp-rsa 公钥加密服务器时间生成 pass
  ↓
POST /index.php/wxApp/sign.html 提交签到
  ↓
服务端返回签到成功或失败
```

## 5. 关键接口

### 5.1 获取学校 Beacon 配置

```text
POST https://wechat.v2.traceint.com/index.php/wxApp/devices.html
```

客户端提交：

```text
页面参数 options，包括 t
```

用途：

```text
服务端根据 t 判断学校/用户/授权上下文
返回该学校允许扫描的 Beacon UUID 列表
```

客户端保存：

```js
uuids = res.data.data.devices
isAuth = true
```

如果没有配置设备，会提示：

```text
当前学校暂未开通蓝牙签到功能
```

### 5.2 获取服务器时间

```text
GET https://wechat.v2.traceint.com/index.php/wxApp/getTime.html
```

用途：

```text
获取服务器时间
用 RSA 公钥加密
生成 pass 参数
```

### 5.3 提交签到

```text
POST https://wechat.v2.traceint.com/index.php/wxApp/sign.html
```

提交参数：

```text
t         授权/学校/用户上下文参数
devices   base64(JSON.stringify([[uuid, major, minor], ...]))
location  base64(JSON.stringify([latitude, longitude]))
pass      RSA(服务器时间)
```

## 6. 参数来源说明

| 参数 | 来源 | 说明 |
|---|---|---|
| `t` | 页面 URL 参数 / 授权页返回 | 用于服务端识别学校、用户或授权上下文 |
| `uuids` | `/devices.html` 响应 | 当前学校允许扫描的 Beacon UUID 列表 |
| `uuid` | `wx.getBeacons()` | 实际扫描到的 Beacon UUID |
| `major` | `wx.getBeacons()` | 实际扫描到的 Beacon major |
| `minor` | `wx.getBeacons()` | 实际扫描到的 Beacon minor |
| `location` | `wx.getLocation()` | 手机 GPS 定位，格式为 `[latitude, longitude]` |
| `serverTime` | `/getTime.html` 响应 | 服务器时间 |
| `pass` | `wxmp-rsa` 加密结果 | `RSA(serverTime)` |

## 7. Beacon 与位置的关系

`[uuid, major, minor]` 是 Beacon 身份标识，不是 GPS 坐标。

```text
uuid   Beacon 大类标识
major  区域 / 场馆 / 分组标识
minor  具体设备 / 小区域标识
```

GPS 位置是单独获取的：

```text
location = [latitude, longitude]
```

因此签到请求中有两份独立数据：

```text
devices   蓝牙 Beacon 身份
location  手机 GPS 位置
```

二者可以由服务端建立间接关系。例如服务端可以配置：

```text
学校 A
  Beacon: uuid + major + minor
  安装位置: 图书馆一楼
  允许 GPS 范围: 某个经纬度半径
```

客户端只负责上传扫描结果，最终是否属于学校、是否在合理范围，应由服务端判断。

## 8. 学校签到参数是否可以变化

### 8.1 `t` 参数可以变化

`t` 来自授权流程，正常设计下应当具备：

```text
临时性
过期时间
用户绑定
学校绑定
防重放能力
```

因此它可以变化，也应该变化。

### 8.2 学校 UUID 列表可以变化

客户端每次通过：

```text
POST /index.php/wxApp/devices.html
```

从服务端获取学校 Beacon UUID 列表。因此服务端可以调整：

```text
新增 Beacon
删除 Beacon
更换 UUID
关闭某学校蓝牙签到
变更学校对应设备
```

### 8.3 `[uuid, major, minor]` 可以变化，但取决于物理设备配置

Beacon 设备本身通常可以配置：

```text
uuid
major
minor
发射功率
广播间隔
```

如果物理设备重新配置，那么 `[uuid, major, minor]` 会变化。但实际运维中通常不会频繁变化，因为需要同步服务端白名单并避免影响正常签到。

### 8.4 `pass` 每次应当变化

`pass` 来源：

```text
GET /getTime.html
  ↓
RSA(serverTime)
```

只要服务器时间变化，`pass` 就会变化。它更像时间签名或防重放参数。

## 9. ESP32 模拟 Beacon 代码的安全含义

提供的 ESP32 代码核心是广播一段 iBeacon 原始数据：

```text
UUID  = FDA50693-A4E2-4FB1-AFCF-C6EB07647825
major = 0x27D7 = 10199
minor = 0xA5EA = 42474
power = 0xC5 = -59
```

这类代码说明：

```text
iBeacon 广播是明文的
uuid/major/minor 可以被现场扫描工具读取
Beacon 身份可以被其他 BLE 设备模拟或重放
```

因此安全上不能只依赖：

```text
客户端扫到了某个 uuid/major/minor
```

如果服务端只校验 Beacon 标识，而不校验其他上下文，就存在被模拟的风险。

## 10. 服务端应校验的内容

最终签到是否成功应由服务端综合判断：

```text
t 是否有效
t 是否过期
t 是否绑定当前用户/学校
Beacon uuid/major/minor 是否属于该学校
Beacon 是否属于允许签到区域
GPS location 是否在合理范围
pass 是否由最近服务器时间生成
pass 是否可重放
同一用户是否重复签到
同一设备是否异常高频签到
是否存在异常 Beacon 或异常定位
```

## 12. 简要结论

当前小程序中的蓝牙签到不是验证码机制，而是：

```text
授权参数 t
+ 服务端下发的学校 Beacon UUID
+ 手机扫描到的 uuid/major/minor
+ 手机 GPS 定位
+ RSA(serverTime) 生成的 pass
```

当前包内没有发现选座验证码实现。Beacon 与 GPS 是两套独立参数，Beacon 标识设备身份，GPS 表示手机位置。学校 Beacon 参数可以由服务端配置变化，物理 Beacon 的 uuid/major/minor 也可重新配置，但通常不会频繁变化。真正的安全性取决于服务端是否严格校验 t、Beacon、GPS、时间窗口和重放风险。
